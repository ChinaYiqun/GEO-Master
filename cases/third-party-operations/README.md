# 第三方 GEO 运营案例与公众号观察

> 收集公开渠道中的 GEO / AI 搜索优化运营案例、公众号文章、服务商方法论与项目复盘，并按照统一证据标准进行拆解。

本目录不是公众号推荐榜，也不代表 GEO-Master 对相关服务、数据或效果作出背书。账号背景、案例结果和运营数据只有在找到原始文章、截图、后台数据或可重复实验后，才会进入“已拆解案例”。

## 目录结构

```text
third-party-operations/
├── README.md                    # 本目录说明与公众号观察池
├── CASE-INDEX.md                # 已拆解及待核验案例索引
├── CASE-SUMMARY-TEMPLATE.md     # 第三方运营案例简版模板
└── cases/                       # 后续逐个案例建立独立目录
    └── source-name-case-name/
```

## 为什么单独建立这个目录

国内 GEO 内容正在快速增长，但公开材料常见三个问题：

1. 方法写得很完整，结果数据却无法独立验证；
2. 把“品牌被提及”“页面被引用”“获得询盘”“最终成交”混成同一个指标；
3. 不说明账号、渠道、内容、广告和销售跟进各自贡献，直接把所有增长归因给 GEO。

本目录会保留案例的传播价值，同时把以下内容拆开：

**案例方做了什么 → 宣称取得什么结果 → 有哪些证据 → 哪些动作可以复制 → 哪些结论不能直接推广**

## 从公众号文章到 GitHub CASE 的处理流程

```text
发现文章或案例
    ↓
保存原始标题、日期、作者和链接
    ↓
提取业务背景、具体动作、工具和周期
    ↓
分离事实、案例方观点与效果宣称
    ↓
建立证据矩阵
    ↓
补充合规风险和其他可能解释
    ↓
形成可复制 SOP 与复现实验
```

### 不直接照搬公众号文章

GitHub 中只保留必要摘要和结构化拆解，不大段复制受版权保护的原文。引用具体观点、数据和截图时，应注明原始来源，并遵守原作者的版权和使用要求。

## 案例进入正式目录的最低要求

一篇公众号文章至少提供以下信息中的四项，才值得建立独立 CASE：

- 明确行业或品牌；
- 明确业务问题；
- 明确实施渠道；
- 明确执行动作；
- 明确时间周期；
- 明确结果指标；
- 存在原始链接、截图或数据；
- 可以设计复现实验。

只有“GEO 很重要”“某企业效果很好”之类的内容，保留在观察池，不进入正式案例库。

## 状态说明

| 状态 | 含义 |
|---|---|
| 待收集 | 只知道账号或案例线索，尚未找到原文 |
| 待核验 | 已找到文章，但未核查关键数据和来源 |
| 拆解中 | 正在整理动作、证据、风险和复现实验 |
| 已发布 | 已形成结构化案例，仍可能继续更新 |
| 有争议 | 关键结论存在冲突或无法复现 |
| 已失效 | 平台规则或案例环境已经发生重大变化 |

## 如何贡献

可以通过 Issue 或 Pull Request 提交：

- 公众号文章链接；
- 公开案例、演讲、访谈或项目复盘；
- 数据截图和可核验结果；
- 对案例归因的质疑；
- 失败案例和平台处罚案例。

提交前请移除客户隐私、登录凭证和未获授权的商业数据。
